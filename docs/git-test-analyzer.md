# Git Test Analyzer

Git Test Analyzer - консольный инструмент для анализа истории git-репозитория с Kotlin-тестами. Он находит тесты, добавленные в коммитах, группирует их по авторам и может сформировать HTML-отчет с динамикой автоматизации.

## Что делает инструмент

- Обходит историю git для Kotlin-файлов (`*.kt`).
- Пропускает merge-коммиты.
- Находит новые тестовые функции с аннотациями `@Test`, `@ParameterizedTest` и `@RepeatedTest`.
- Определяет автора теста по e-mail автора коммита.
- Определяет дату добавления теста по дате коммита.
- Извлекает `@System("...")` на уровне класса или конкретного теста.
- Убирает дубли по имени тестовой функции, оставляя самую позднюю запись.
- Печатает консольный отчет по авторам.
- При включенной настройке генерирует HTML-отчет.

## Требования

- JDK 17 или новее.
- Maven.
- Локальный git-репозиторий, который нужно анализировать.
- Анализируемый репозиторий должен содержать Kotlin-файлы с JUnit-тестами.

## Быстрый старт

1. Откройте `src/main/kotlin/com/analyzer/AnalyzerConfig.kt`.
2. Укажите путь к анализируемому репозиторию:

```kotlin
val REPO_PATH = "/path/to/your/test-repo"
```

3. При необходимости настройте период консольного отчета:

```kotlin
val DAYS: Int? = 7
```

`null` означает отчет за все время, `7` - за последнюю неделю, `30` - за последний месяц.

4. Запустите инструмент:

```bash
mvn compile exec:java
```

5. Если `GENERATE_HTML = true`, откройте HTML-отчет:

```text
test-report/report.html
```

Это единый автономный файл: все стили, скрипты (включая Chart.js) и данные
встроены inline. Файл можно отправить любому — он откроется в браузере
двойным кликом без доступа к сети и без внешних зависимостей.

## Основные команды

| Команда | Назначение |
| --- | --- |
| `mvn test` | Запустить тесты проекта |
| `mvn compile exec:java` | Запустить анализатор через Maven |
| `mvn package` | Собрать jar проекта |

Собранный `jar` не является fat jar: зависимости Kotlin не упаковываются внутрь. Для обычного запуска удобнее использовать `mvn compile exec:java`.

## Настройки

Все настройки находятся в `AnalyzerConfig.kt`.

| Настройка | Описание |
| --- | --- |
| `REPO_PATH` | Абсолютный или относительный путь к анализируемому git-репозиторию |
| `DAYS` | Период для консольного отчета в днях; `null` - все время |
| `GENERATE_HTML` | Включает или выключает генерацию HTML-отчета |
| `HTML_REPORT_DIR` | Директория, куда будет записан HTML-отчет |
| `THREAD_COUNT` | Количество потоков для параллельного анализа коммитов |
| `SYSTEM_NAMES` | Словарь для замены ID системы на человекочитаемое название в HTML-отчете |
| `AUTHOR_NAMES` | Словарь для объединения нескольких e-mail одного автора в HTML-отчете |

Пример настройки названий систем:

```kotlin
val SYSTEM_NAMES: Map<String, String> = mapOf(
    "CI01337" to "Платежная система",
    "CI02000" to "Система авторизации",
)
```

Пример объединения e-mail одного автора:

```kotlin
val AUTHOR_NAMES: Map<String, String> = mapOf(
    "ivanov@company.com" to "Иванов Иван",
    "ivan.ivanov@company.com" to "Иванов Иван",
)
```

Важно: `AUTHOR_NAMES` и `SYSTEM_NAMES` используются и в HTML-отчете, и в консольном отчёте. 
В консольном отчёте автор показывается как «Имя (email)», если e-mail зарезолвлен через словарь, 
а в детальном списке рядом с названием теста показывается метка системы: `[Название (ID)]` 
(если задано `SYSTEM_NAMES`) или `[ID]` иначе.

## Как данные собираются с коммитов

Сбор данных идет через локальный `git` в репозитории из `AnalyzerConfig.REPO_PATH`. Инструмент не обращается к удаленному репозиторию, не использует GitHub/GitLab API и не читает текущие незакоммиченные изменения как отдельный источник данных.

### 1. Проверка репозитория

Перед анализом выполняется:

```bash
git -C "$REPO_PATH" rev-parse --is-inside-work-tree
```

Если команда не возвращает `true`, запуск останавливается.

### 2. Получение списка коммитов

Список коммитов формируется командой:

```bash
git -C "$REPO_PATH" log --format=%H%x00%aE%x00%aI --no-merges -- "*.kt"
```

Из каждой строки берутся три поля, разделённые NUL (`%x00`):

| Поле | Источник | Куда попадает |
| --- | --- | --- |
| hash | `%H` | `CommitInfo.hash` |
| e-mail автора | `%aE` | `CommitInfo.authorEmail`, затем `TestRecord.authorEmail` |
| дата автора | `%aI` | `CommitInfo.date`, затем `TestRecord.date` |

Важные следствия:

- анализируются только коммиты, которые затрагивают Kotlin-файлы по pathspec `*.kt`;
- merge-коммиты исключаются флагом `--no-merges`;
- используется author date (`%aI`), а не committer date;
- автор теста определяется по author e-mail коммита, без `git blame`.

### 3. Подготовка root-коммитов

Отдельно вычисляется набор root-коммитов:

```bash
git -C "$REPO_PATH" rev-list --max-parents=0 HEAD
```

Это нужно, чтобы для начального коммита запрашивать diff с флагом `--root`; без него у root-коммита нет родителя, с которым можно сравнить дерево.

### 4. Получение diff каждого коммита

Для каждого коммита запрашивается patch только по Kotlin-файлам:

```bash
git -C "$REPO_PATH" diff-tree -M -U999999 -p "$COMMIT_HASH" -- "*.kt"
```

Для root-коммита команда дополняется `--root`:

```bash
git -C "$REPO_PATH" diff-tree --root -M -U999999 -p "$COMMIT_HASH" -- "*.kt"
```

Параметры важны для качества данных:

| Параметр | Зачем нужен |
| --- | --- |
| `-p` | вернуть patch/diff, который можно разобрать построчно |
| `-M` | включить detection переименований, чтобы перенос файла не выглядел как полное удаление и добавление |
| `-U999999` | дать почти полный контекст файла, чтобы парсер видел `@System` на классе даже далеко от добавленного теста |
| `-- "*.kt"` | ограничить анализ Kotlin-файлами |

Коммиты обрабатываются параллельно через `Dispatchers.IO.limitedParallelism(THREAD_COUNT)`. Чтобы не запускать слишком много git-процессов одновременно, коммиты делятся на пакеты размером `THREAD_COUNT * 4`.

### 5. Разбор diff

`TestParser.findNewTests()` читает diff построчно.

Парсер определяет текущий файл по строкам вида:

```diff
+++ b/path/to/FileTest.kt
```

Затем внутри каждого hunk-а он отдельно собирает:

- добавленные тестовые функции из строк `+`;
- удаленные тестовые функции из строк `-`;
- контекстные строки без префикса, чтобы понимать класс, вложенность и унаследованный `@System`.

Новым тестом считается добавленная функция с одной из поддерживаемых JUnit-аннотаций:

- `@Test`
- `@ParameterizedTest`
- `@RepeatedTest`

Поддерживаются два варианта записи:

```kotlin
@Test
fun shouldWork() {
}
```

```kotlin
@Test fun shouldWork() {
}
```

Между тестовой аннотацией и `fun` могут идти другие аннотации, например `@DisplayName` или `@System`.

Имя функции извлекается регулярным выражением для `fun ...(`. Поддерживаются обычные Kotlin-имена и имена в backticks:

```kotlin
fun shouldWork()
fun `should work with spaces`()
```

Если в том же hunk-е функция с таким же именем есть среди удаленных строк, добавление не считается новым тестом. Так инструмент снижает шум от переносов, рефакторингов и изменений тела теста, где старое объявление удалили и новое объявление добавили обратно с тем же именем.

### 6. Извлечение `@System`

Во время разбора diff парсер ищет только строковую форму:

```kotlin
@System("CI01337")
```

Поддерживаются два уровня:

- `@System` перед `class` или `object` применяется ко всем тестам внутри этого контейнера;
- `@System` рядом с конкретным тестом переопределяет систему класса.

Парсер ведет стек классов по глубине фигурных скобок. Скобки внутри строковых литералов не учитываются. Вложенные классы наследуют `@System` родительского класса, если у них нет собственной аннотации. `companion object` не считается отдельным тестовым контейнером.

Когда тест найден, парсер возвращает `NewTestInfo`:

| Поле | Как определяется |
| --- | --- |
| `functionName` | имя найденной тестовой функции |
| `filePath` | путь из `+++ b/...` |
| `systemId` | `@System` теста, класса или унаследованного класса; иначе `null` |

### 7. Превращение найденных тестов в записи отчета

В `Main.kt` каждый `NewTestInfo` объединяется с метаданными коммита:

| `TestRecord` | Источник |
| --- | --- |
| `authorEmail` | e-mail автора коммита из `%aE` |
| `functionName` | результат парсера diff |
| `filePath` | результат парсера diff |
| `date` | author date коммита из `%aI` |
| `systemId` | результат парсера diff |

То есть одна запись отчета означает: "в этом коммите автор с таким e-mail добавил тестовую функцию с таким именем в таком Kotlin-файле".

### 8. Второй проход для пропущенных `systemId`

Если у найденного теста `systemId == null`, инструмент пытается дополнить систему отдельным проходом по истории файла.

Для каждого файла с такими записями выполняется pickaxe-поиск коммитов, где менялась строка `@System(`:

```bash
git -C "$REPO_PATH" log --all --reverse --format=%H -S "@System(" -- "$FILE_PATH"
```

Для каждого найденного коммита читается полное содержимое файла на тот момент:

```bash
git -C "$REPO_PATH" show "$COMMIT_HASH:$FILE_PATH"
```

Затем `TestParser.extractSystemMapping()` строит отображение:

```text
functionName -> systemId
```

Если имя теста из исходной записи найдено в этом отображении, `systemId` в `TestRecord` дополняется. Это покрывает сценарий, когда тест добавили раньше, а `@System` на класс или тест проставили позже.

### 9. Постобработка перед отчетами

После сбора всех записей выполняются три операции:

1. `deduplicateLatestTests()` сортирует записи по дате коммита от новых к старым и оставляет только первую запись для каждого `functionName`.
2. Для консольного отчета `filterRecordsWithinDays()` применяет период из `AnalyzerConfig.DAYS`.
3. `buildTestsByAuthor()` группирует записи по исходному `authorEmail`.

HTML-отчет получает все дедуплицированные записи без фильтра `DAYS`; его периодические фильтры применяются уже в браузере.

## Как определяется новый тест

Инструмент получает diff каждого подходящего коммита через `git diff-tree` с полным контекстом файла. Затем парсер ищет добавленные тестовые функции.

Поддерживаются такие формы:

```kotlin
@Test
fun shouldCreateOrder() {
    // ...
}

@ParameterizedTest
@ValueSource(strings = ["a", "b"])
fun shouldValidateInput(value: String) {
    // ...
}

@RepeatedTest(3)
fun shouldRetryOperation() {
    // ...
}
```

Также поддерживается форма, где аннотация и функция находятся на одной строке:

```kotlin
@Test fun shouldWork() {
    // ...
}
```

Если в одном изменении тестовая функция одновременно удалена и добавлена с тем же именем, она не считается новой. Это снижает шум от переносов и рефакторингов.

## Как определяется система

Инструмент ищет аннотацию `@System("...")`.

Аннотация на классе применяется ко всем тестам внутри класса:

```kotlin
@System("CI01337")
class PaymentTest {
    @Test
    fun shouldPay() {
        // ...
    }
}
```

Аннотация на тесте переопределяет систему класса:

```kotlin
@System("CI01337")
class PaymentTest {
    @Test
    @System("CI02000")
    fun shouldAuthorizeBeforePayment() {
        // ...
    }
}
```

Вложенные классы наследуют `@System` родительского класса, если у вложенного класса нет собственной аннотации. `companion object` не считается отдельным тестовым контейнером.

Если тест был добавлен до появления `@System` на классе, инструмент выполняет второй проход по истории файла и пытается дополнить отсутствующий `systemId` по коммитам, где добавлялась или менялась аннотация `@System`.

## Консольный отчет

Консольный отчет показывает:

- путь к анализируемому репозиторию;
- период;
- количество новых тестов по авторам;
- общий итог;
- список найденных тестов по каждому автору.

Период консольного отчета задается настройкой `DAYS`. Фильтр применяется после удаления дублей.

## HTML-отчет

Если включено `GENERATE_HTML`, инструмент создает директорию из `HTML_REPORT_DIR`
и кладет в нее единственный файл `report.html`:

```text
test-report/
  report.html
```

Это автономный самодостаточный файл: все стили (CSS), скрипты (включая Chart.js)
и данные отчёта встроены inline. Файл можно отправить любому получателю —
он откроется в браузере двойным кликом без доступа к сети и без внешних
зависимостей.

HTML-отчет содержит:

- сводку по авторам;
- график тестов по авторам;
- динамику по времени;
- таблицу и графики по системам;
- фильтр по системе;
- фильтры по периодам: неделя, месяц, квартал, год, конкретный месяц, все время;
- детальный список найденных тестов.

Для HTML-отчета в `report.html` встраиваются все найденные дедуплицированные
записи, а фильтры применяются на стороне браузера.

## Ограничения и важные детали

- Анализируются только файлы `*.kt`.
- Merge-коммиты не анализируются.
- Дедупликация выполняется только по имени функции. Если в разных файлах есть тесты с одинаковым именем, останется самая поздняя запись.
- Переименованный тест может быть интерпретирован как новый, если старое имя было удалено, а новое добавлено.
- Инструмент полагается на историю git. Squash, rebase и переписанная история могут изменить результаты.
- Поддерживается только синтаксис `@System("...")` со строковым литералом.
- Парсер детектит обычные, backtick, extension (`fun Receiver.foo`) и generic (`fun <T> foo`, `fun foo<T>`) функции. Extension-функции с generic-типом receiver (`fun List<Int>.foo()`) не поддерживаются — парсер возьмёт имя типа.
- Поддерживаются все виды типов-контейнеров (`class`, `data class`, `sealed class`, `enum class`, `inner class`, `value class`, `object`); `companion object` не считается отдельным контейнером.
- Консольный отчёт фильтрует за «последние N дней» = ровно N календарных дней включая сегодня (`days=7` даёт окно из 7 дней, а не 8).

## Окружение git

- Все обращения к git выполняются с `GIT_PAGER=cat` и `GIT_TERMINAL_PROMPT=0`, чтобы pager и интерактивные запросы учётных данных не блокировали чтение потоков.
- Вывод git читается в UTF-8; исходные файлы ожидаются в UTF-8 (см. `AGENTS.md`).
- Часовой пояс метки `Сформирован` в HTML-отчёте по умолчанию берётся системный (`ZoneId.systemDefault()`).

## Логирование

Уровень логирования задаётся системным свойством `-Danalyzer.log=DEBUG|INFO|WARN|ERROR` (по умолчанию `INFO`). Повторяющиеся предупреждения от git-процессов подавляются; в конце прогона печатается суммарный счётчик.

## Когда результат может отличаться от ожиданий

- Тест добавлен не как функция с поддерживаемой JUnit-аннотацией.
- Тест находится не в Kotlin-файле.
- Автор коммита не совпадает с фактическим автором теста.
- В одном проекте часто используются одинаковые имена тестовых функций.
- `@System` задан через константу или другую форму, отличную от `@System("ID")`.
